# 面试讲稿

## 1. 为什么做这个项目？

我之前的经历更多集中在金融审计和经营数据整理，但我希望投递的岗位也包括数据分析、商业分析、产品分析和运营分析。因此我补充了一个更通用的用户行为分析项目，重点展示 SQL 指标加工、Python 分析、漏斗和留存方法，以及把数据结论转化成产品/运营建议的能力。

## 2. 数据来源是什么？

我使用的是公开匿名化电商行为数据，原始来源是 REES46 / Kaggle 的 multi-category ecommerce behavior dataset。数据包含用户浏览、加购、购买等事件，以及商品、价格、品牌、用户和 session 字段。为了避免下载完整大文件，我通过 Hugging Face mirror 的 rows API 拉取样本做本地复现。

## 3. 为什么不用虚拟数据？

虚拟数据很难说服面试官，因为分布和行为逻辑可能是人为设定的。这个项目使用真实公开日志，虽然不是企业内部数据，但字段结构和分析方法接近真实产品数据分析场景，更适合展示指标体系、SQL 查询、漏斗分析和 dashboard 设计能力。

## 4. 前后端怎么分？

- 后端：`data_source` 拉取公开数据，`prepare_data` 清洗并生成样本，`build_database` 生成 DuckDB 指标库，`sql/` 和 `advanced_analysis` 负责指标和深度分析。
- 前端：`app/dashboard.py` 用 Streamlit 展示核心指标、漏斗、留存、路径、分层和业务洞察，并提供日期、品类、品牌和价格筛选。

## 5. 你分析了什么指标？

- DAU、session 数、浏览量、加购数、购买数和 GMV。
- 浏览到加购、加购到购买的转化漏斗。
- D0/D1/D3/D7 cohort 留存。
- session 内 Top 用户路径。
- 浏览型、加购未买型、购买型和复购型用户分层。
- 价格带、品类、session 深度和购买路径差异。

## 6. 发现了什么问题？

当前样本里，大部分用户停留在浏览阶段，浏览到加购的流失明显；加购后仍有部分用户没有购买。说明优化重点不是单纯提高访问量，而是提升浏览后的行动意愿，并对加购用户进行更及时的转化触达。

## 7. 如何转化为建议？

- 对只浏览用户：优化首页推荐、商品详情页信任信息和首单引导。
- 对加购未买用户：增加价格提醒、优惠券触达和购物车召回。
- 对购买用户：设计复购提醒、会员权益和高相关品类推荐。
- 对高 GMV 品类：倾斜推荐位和活动资源。

## 一句话总结

这个项目用公开电商行为日志模拟真实产品数据分析流程，从 SQL 指标加工到 Python 漏斗、留存、路径分析，再到 Streamlit dashboard 和业务建议，体现的是我把原始用户行为数据转化为产品运营决策的能力。
